# Task Manager CLI – Technical Design

## 1. Data Models

### Task

The sole entity in the application. Stored in a plain JavaScript `Map` keyed by `id`.

| Property | Type | Required | Validation rules |
|---|---|---|---|
| `id` | `string` | yes (auto) | UUID v4 generated by `crypto.randomUUID()`; never user-supplied |
| `title` | `string` | yes | Non-empty; max 200 characters; leading/trailing whitespace stripped |
| `description` | `string` | no | Defaults to `''`; max 1000 characters |
| `status` | `string` | yes (auto) | One of `todo`, `in-progress`, `done`; defaults to `todo` on creation |
| `priority` | `string` | yes (auto) | One of `low`, `medium`, `high`; defaults to `medium` on creation |
| `createdAt` | `string` | yes (auto) | ISO 8601 timestamp; set once at creation; never modified |
| `updatedAt` | `string` | yes (auto) | ISO 8601 timestamp; set at creation; updated on every `updateTask` call |

### Validation Rules

Rules are enforced by `src/validator.js` and re-checked defensively inside `TaskManager` before any mutation.

#### `id`
- Must not be supplied by the caller on `addTask`; the field is ignored if present.
- On `updateTask` and `deleteTask`, must be a non-empty string matching the UUID v4 pattern: `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`.
- Error message: `'id must be a valid UUID v4 string'`

#### `title`
- Must be a `string` type; non-string values throw immediately.
- Leading and trailing whitespace is stripped before validation and storage (`.trim()`).
- After trimming, length must be ≥ 1 character.
- After trimming, length must be ≤ 200 characters.
- Error messages:
  - `'title is required'` — when absent, empty, or whitespace-only
  - `'title must not exceed 200 characters'` — when trimmed length > 200

#### `description`
- Optional. When omitted or `undefined`, stored as `''`.
- When supplied, must be a `string` type.
- Leading and trailing whitespace is stripped before storage (`.trim()`).
- After trimming, length must be ≤ 1000 characters.
- Error message: `'description must not exceed 1000 characters'`

#### `status`
- Must be one of the exact strings: `'todo'`, `'in-progress'`, `'done'`.
- Case-sensitive; `'TODO'` or `'In-Progress'` are rejected.
- Default on creation: `'todo'`.
- Cannot be set to any other value on `updateTask`.
- Error message: `"status must be one of: todo, in-progress, done"`

#### `priority`
- Must be one of the exact strings: `'low'`, `'medium'`, `'high'`.
- Case-sensitive; `'Low'` or `'HIGH'` are rejected.
- Default on creation: `'medium'`.
- Cannot be set to any other value on `updateTask`.
- Error message: `"priority must be one of: low, medium, high"`

#### `createdAt`
- Set automatically to `new Date().toISOString()` at creation.
- Read-only — any attempt to set it via `updateTask` is silently ignored (the field is stripped from the updates object before merging).

#### `updatedAt`
- Set automatically to `new Date().toISOString()` at creation, matching `createdAt`.
- Automatically refreshed to `new Date().toISOString()` on every successful `updateTask` call.
- Read-only from the caller's perspective — user-supplied values are stripped from updates the same way as `createdAt`.

## 3. Module Responsibilities

### `src/taskManager.js`

**Exports:** `TaskManager` class (default export)

**Responsibilities:**
- Maintains the in-memory `Map<string, Task>` store.
- Provides all CRUD operations: `addTask`, `getTaskById`, `getAllTasks`, `updateTask`, `deleteTask`.
- Provides query helpers: `filterByStatus`, `filterByPriority`, `sortByPriority`, `sortByDate`.
- Calls `validator.js` helpers before mutating state (defense-in-depth).
- Throws `NotFoundError` when a task ID is not present in the store.

**Dependencies:** `src/validator.js`

```js
// Exported API surface
export class TaskManager {
  addTask({ title, description })
  getTaskById(id)
  getAllTasks()
  updateTask(id, updates)
  deleteTask(id)
  filterByStatus(status)
  filterByPriority(priority)
  sortByPriority(tasks)
  sortByDate(tasks, order)   // order: 'asc' | 'desc'
}
```

---

### `src/validator.js`

**Exports:** named pure functions

**Responsibilities:**
- Validates all user-supplied values before they reach `TaskManager`.
- Throws `ValidationError` with a descriptive message on the first failing rule.
- Exports individual validators so `cli.js` can run them before dispatching.

**Dependencies:** none

```js
// Exported API surface
export function validateTitle(title)
export function validateDescription(description)
export function validateStatus(status)
export function validatePriority(priority)
export function validateId(id)
export function validateSortOrder(order)
export function validateSortBy(field)
```

---

### `src/formatter.js`

**Exports:** named pure functions

**Responsibilities:**
- Converts a `Task` object or array into a printable string.
- `formatTable` renders an ASCII table with columns: ID, Title, Status, Priority, CreatedAt.
- `formatJson` renders newline-delimited JSON.
- `formatSuccess` renders a consistent `✓ [Operation] successful` confirmation with task details.

**Dependencies:** none

```js
// Exported API surface
export function formatTable(tasks)
export function formatJson(tasks)
export function formatSuccess(operation, task)
```

---

### `src/cli.js`

**Exports:** none (executable entry point)

**Responsibilities:**
- Parses `process.argv` using the built-in `util.parseArgs` (Node 18+).
- Routes to the correct `TaskManager` method based on the subcommand.
- Calls `validator.js` to validate arguments before delegating.
- Catches all thrown errors, logs them with `console.error`, and exits with the appropriate code.
- Prints output using `formatter.js`.

**Dependencies:** `src/taskManager.js`, `src/validator.js`, `src/formatter.js`

**Supported commands:**

| Command | Arguments / Flags |
|---|---|
| `create` | `--title <str>` (required), `--description <str>` (optional) |
| `list` | `--filter-status <val>`, `--filter-priority <val>`, `--sort-by <val>`, `--sort-order <val>` |
| `update` | `--id <uuid>` (required), `--title`, `--description`, `--status`, `--priority` |
| `delete` | `--id <uuid>` (required) |
| `--help` | Print usage information |

## 4. Error Handling Strategy

### Error types

All custom error classes extend the built-in `Error`. They are defined inline in the module that throws them; no separate error file is needed at this scope.

| Class | Thrown by | Cause | Process exit code |
|---|---|---|---|
| `ValidationError` | `validator.js`, `taskManager.js` | A field value fails a validation rule | `1` |
| `NotFoundError` | `taskManager.js` | No task exists for the given ID | `2` |
| `InvalidCommandError` | `cli.js` | Unknown subcommand | `3` |
| `InvalidArgumentError` | `cli.js` | Unrecognised or malformed CLI flag | `4` |

### Error flow

```
cli.js  ──validates args──▶  validator.js  (throws ValidationError / InvalidArgumentError)
   │
   └──dispatches──▶  taskManager.js  (throws ValidationError / NotFoundError)
                           │
                           └──re-thrown up to cli.js catch block
                                    │
                                    └── console.error(err.message)
                                         process.exit(code)
```

### Rules (consistent with `.github/copilot-instructions.md`)

- All thrown values are `Error` instances, never plain strings.
- `cli.js` wraps the entire dispatch in a `try/catch`; individual module functions also use `try/catch` where I/O or third-party calls could fail.
- Errors are logged with `console.error`, never `console.log`.
- Success output is written to `console.log` / `process.stdout`.
